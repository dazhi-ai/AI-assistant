## 一、播歌打断逻辑修复（已完成并验证）

### 背景与目标

- 现象：要求播放歌曲时，偶尔在"找到歌曲准备播放"时进入聆听状态，导致没有声音；个别歌曲出现口令完全不响应、卡死。
- 目标：找到歌曲后**直接播放**；播放期间噪声不误打断；**口令（唤醒词）可随时打断/换歌**。

### 根因定位（依赖线上日志逐轮分析）

1. **首句缺 `tts start`**：音乐首句前偶尔没有补发 `tts start`，设备停留在聆听态，听不到音乐。
   - 深层原因是竞态：原本用 `netease_music_wait_first_downlink` 条件触发 `tts start`，但播报 TTS 的残留 Opus 帧会提前把该标志清掉，导致音乐首句到达时 `tts start` 被跳过。
2. **3 秒"绝对防打断"误伤唤醒词导致死锁**：原逻辑在首帧后 3 秒内忽略一切 abort（含 `wake_word_detected`）。
   - 但设备固件一旦本地识别到唤醒词就**立即停播、进入聆听**，这一步服务端管不了。
   - 当唤醒词落在 3 秒硬窗口内被服务端"忽略"时：设备已停播在听，服务端却以为还在放歌、继续发帧不响应 → **两边状态错位、死锁**。
   - 线上复现："落（花开花落日升日没）"这首，唤醒落在硬窗口末 0.3s 被忽略，之后约 2.5 分钟服务端零日志（彻底卡死）。

### 修复方案

| 文件 | 改动 |
| --- | --- |
| `scripts/xiaozhi/plugins_func/functions/play_music_netease.py` | 引入独立标志 `netease_music_need_tts_start`，在入队首句**之前**置位，且只由音乐首句消费，避免被播报残帧提前清掉；防打断窗口覆盖整首估算时长。 |
| `scripts/xiaozhi/patches/sendAudioHandle.py` | 处理音乐 `SentenceType.FIRST` 时，依据 `need_tts_start` 可靠补发 `tts start`，让设备从聆听态切回播放态。 |
| `scripts/xiaozhi/patches/abortHandle.py` | **3 秒绝对防打断改为只拦截"非唤醒"abort（噪声/VAD）**；唤醒词任何时候都立即放行打断。修复了唤醒词被忽略导致的死锁。 |
| `scripts/xiaozhi/patches/abortMessageHandler.py` | 移除原 1.5s 去抖，纯透传给 `handleAbortMessage`，不干扰分层打断逻辑。 |

### 最终行为（验证通过）

- 找到歌曲 → **直接播放**。
- **单次唤醒词随时可打断 / 换歌**（不再有 3 秒延迟，因为延迟会触发死锁）。
- 环境噪声 / VAD → 整首播放期间忽略，不误打断。

### 重要取舍

> 用户最初期望"播放 3 秒内连唤醒词也压住不打断"，但该需求与设备固件天然冲突——设备一识别到唤醒词就本地停播了，服务端无法让它继续放。因此唤醒词必须始终立即响应；"3 秒保护"现在只对噪声有效。

## 二、新闻联网搜索方案（已定方案，待前置依赖）

### 现状痛点

当前新闻是**离线灌入 system prompt**：OpenClaw 定时生成简报 → `merge_system_prompt_for_daily_news.py` 拼接 → 写入 MySQL `ai_agent.system_prompt`。

痛点：不实时、prompt 臃肿吃 token、运维链路重、无法按需触发。

### 调研结论：火山方舟 Web Search（联网内容插件）

- 联网搜索是 **Web Search（联网内容插件）**，需绑定到控制台创建的 **Bot / 零代码应用**，开启联网插件后拿到 `bot_id`。
- 调用走 **Bots 端点**（`/api/v3/bots/chat/completions`，OpenAI 兼容，`model=bot_id`）或 Responses API（`FunctionName=webSearch` + `ParamsString` 带 `bot_id`）。
- 触发模式：**总是开启** vs **按需开启**（推荐按需，省钱省延迟）。
- 需在控制台单独开通插件，按调用计费（有免费额度）。
- 注："Coding Plan / Agent Plan" 只是方舟文档/套餐分区命名，功能本体在「工具调用 → Web Search」。

### 已确定的落地决策

- **方案 A**：新增 `get_news` 本地插件（仿 `get_weather`/`play_music` 的 `register_function`），实时、按需，改动最小。
- **后端**：复用现有 ARK Key，调用火山方舟联网 Bot。
- **旧链路**：保留 system prompt 新闻段作为**兜底**（联网失败/超时时回退）。
- 主对话 LLM 是 `DoubaoLLM`（type=openai，`ark.cn-beijing.volces.com/api/v3`，`doubao-1-5-pro-32k`），意图走 `function_call`。
- ARK Key 由智控台（manager-api / MySQL）下发，本地 yaml 为空；接插件时直接复用，无需手填。
