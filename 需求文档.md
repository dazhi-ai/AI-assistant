# AI助手项目需求规格说明书（SRS）

## 1. 文档信息
- 项目名称：AI助手
- 文档类型：需求规格说明书（SRS）
- 版本：v1.0
- 编写日期：2026-02-25
- 目标读者：产品、后端、前端、硬件、测试、运维

## 2. 项目背景与目标
本项目将原有依赖官方云的方案迁移为自建后端中枢，实现“语音交互 + Live2D 视觉 + 网易云音乐控制 + 生活服务”的一体化 AI 助手。

### 2.1 架构原则
- 所有第三方能力（豆包、网易云、天气）统一由自建后端代理与编排。
- 平板前端仅连接自建后端，不直接访问第三方 API。
- 小智硬件、平板、第三方服务之间通过后端形成唯一安全网关。

### 2.2 主链路
- 指令链路：小智硬件 -> Python 后端 -> 豆包 API -> Python 后端
- 控制链路：Python 后端 -> 平板 Web（Live2D）/ 网易云服务
- 反馈链路：豆包流式文本 -> TTS 音频流 -> 平板播放 + Live2D 口型驱动

## 3. 范围定义
### 3.1 本期范围（In Scope）
- 自建 Python 后端与 WebSocket 网关
- 豆包模型接入（含 Function Calling）
- 平板端 Live2D 展示与表情/口型联动
- 网易云音乐控制插件（登录、搜索、播放、点红心、收藏、歌单）
- 天气查询与 3 天预报卡片展示
- 安全基线与部署规范

### 3.2 非本期范围（Out of Scope）
- 多端账号系统与用户中心
- 公网商用级高可用（多机房容灾）
- 复杂推荐算法与音乐个性化排序

## 4. 角色与使用场景
- 终端用户：通过语音与 AI 交互，控制音乐、切换形象、查询天气。
- 设备终端：小智硬件负责采集语音；平板负责视觉与音频播放。
- 运维/开发：部署后端服务、管理密钥、监控运行状态。

## 5. 功能需求

### FR-001 视觉形象与人设管理（Live2D）
1. 系统应支持通过 System Prompt 配置 AI 人设（性格、口头禅、语气）。
2. 系统应支持语音触发形象切换（示例：“换成运动风”）。
3. 后端应向平板推送模型切换 JSON 指令，前端据此加载对应 Live2D 模型。
4. 人设文本与视觉形象应保持一致性，不出现风格冲突。

### FR-002 流式语音响应与低延迟播放
1. 后端应接收豆包流式文本并实时触发 TTS 合成。
2. TTS 音频应通过 WebSocket 分片推送至平板，支持边下发边播放。
3. 平板应使用 Web Audio API 实时提取频率或振幅特征，驱动 Live2D 口型参数（Mouth Open）。
4. 播放过程中应有缓冲管理机制，降低卡顿与断音概率。

### FR-003 网易云音乐深度交互（Function Calling）
1. 系统应支持豆包工具调用（tool_calls）触发后端音乐函数执行。
2. 登录管理：后端生成网易云登录二维码 URL，并通过 WebSocket 推送到平板浮窗展示。
3. 点红心/收藏：
   - 用户说“好听”时，执行 `like_music(song_id)`（点红心）。
   - 用户说“收藏”时，执行收藏动作（与点红心语义区分）。
4. 搜索播放：执行 `search_music(keywords)`；若存在多个候选，AI 应反问用户选择。
5. 歌单播放：执行 `get_user_playlist(type)`，支持“我喜爱的/我收藏的”随机播放。
6. 可视反馈：关键动作成功后，后端可推送效果消息（如 HEART 动画）给平板端。

### FR-004 生活服务（天气）
1. 用户查询天气时，系统应提供语音播报。
2. 平板端应同步展示和风天气 3 天预报卡片/图表。

### FR-005 模型参数动态调优
1. 后端应支持按场景动态调整 Temperature。
2. 建议策略：事实查询低 Temperature；闲聊和形象互动高 Temperature。

## 6. 非功能需求

### NFR-001 性能与时延
- 语音交互应采用流式链路，减少首包等待时间。
- TTS 推流与前端播放应支持分片缓冲，避免明显停顿。

### NFR-002 兼容性
- 平板端需兼容老旧 Android WebView（小米平板 1）。
- 前端代码应通过 Babel 编译为 ES5。
- 建议实现基于 `setInterval` 的心跳保活，降低静置掉线概率。

### NFR-003 安全性
- 所有密钥必须存储在 `.env`，禁止硬编码到仓库。
- 网易云 API 服务端口仅本地开放（如 `127.0.0.1:3000`）。
- WebSocket 连接必须进行 token 校验（如 `?token=xxx`）。
- 网易云登录 Cookie 仅后端保存于受保护目录，不下发前端。

### NFR-004 可运维性
- 服务应支持守护进程与开机自启（采用 systemd）。
- 应具备基本日志能力（请求、工具调用、异常、重连）。

## 8. 系统模块划分（WBS）

### 阶段一：安全后端与模型中枢
- 任务 1.1：部署网易云 API（使用 Node.js 进程 + systemd 托管）
- 任务 1.2：部署小智后端并接入豆包模型
- 任务 1.3：完成 `.env` 脱敏与密钥治理
- 任务 1.4：开发 Function Calling 解析器与网易云插件

### 阶段二：平板视觉容器
- 任务 2.1：接入 Live2D 模型渲染
- 任务 2.2：实现后端消息驱动（模型切换、效果反馈、二维码弹窗）
- 任务 2.3：完成 ES5 兼容改造与心跳保活

### 阶段三：硬件接入与联调部署
- 任务 3.1：烧录 ESP32 固件并指向自建后端
- 任务 3.2：全链路联调（语音、音乐、天气、视觉）
- 任务 3.3：安全自检与上线部署

## 9. 里程碑计划
- M1：本地跑通网易云 API 与豆包 API 调用
- M2：平板成功展示 Live2D 模型
- M3：打通 WebSocket，实现“输入文本 -> 平板语音+口型”
- M4：接入小智硬件麦克风，替代手工输入
- M5：完成安全加固、自检与部署上线

## 10. 验收标准（UAT）
1. 语音指令可稳定触发对应功能（聊天、换造型、天气、音乐控制）。
2. “好听/收藏”等关键词可正确映射到不同网易云动作。
3. 平板端可展示登录二维码、天气卡片、Live2D 口型联动。
4. 项目源码不包含明文密钥；WebSocket 具备 token 校验。
5. 服务重启后可自动恢复运行。

## 11. 风险与应对
- 老旧平板 WebGL/内核兼容风险：优先 ES5 构建，必要时尝试 Crosswalk 或轻量浏览器模式。
- 流式音频卡顿风险：优化分片大小与前端缓冲策略。
- 第三方 API 波动风险：增加超时、重试、降级与错误提示机制。
